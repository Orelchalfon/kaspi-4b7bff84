# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

`kaspii-web` is the TanStack Start + Supabase web app for **KidCoin / Kaspi** ‚Äî a Hebrew-first RTL family-finance app where parents assign tasks, approve submissions, and children earn coins they can spend, save (via an automatic savings %), or route into goals. The parent `../CLAUDE.md` covers the broader monorepo; this file is web-specific.

`.lovable/plan.md` is the original feature spec for savings + goals. As of `20260518185128_add_savings_and_goals.sql` the DB pieces are in place and the child `/child/savings` route is wired up.

## Commands

- `pnpm install` ‚Äî install deps (pnpm is the only supported package manager)
- `pnpm dev` ‚Äî Vite dev server on `http://localhost:8080`
- `pnpm build` / `pnpm build:dev` ‚Äî production build (uses `@cloudflare/vite-plugin`, target = Cloudflare Workers)
- `pnpm preview` ‚Äî preview the production build
- `pnpm lint` ‚Äî ESLint (flat config, TS + react-hooks + react-refresh + prettier)
- `pnpm format` ‚Äî Prettier (see `.prettierrc`: 100-col, semi, double quotes, trailing-comma all)
- `pnpm test` ‚Äî Vitest **integration tests** (`tests/**/*.test.ts`, node env, single-fork, 60s timeout)
- `pnpm test:e2e` ‚Äî same as above but only `tests/e2e/`
- Run a single test file: `pnpm vitest run tests/e2e/parent-child-flow.test.ts`
- Run by name: `pnpm vitest run -t "parent creates household"`
- Typecheck only: `pnpm exec tsc --noEmit`

> Tests hit the **real** Supabase project (`flxhxmrtdqegfsupvvus`) ‚Äî they create ad-hoc auth users via the service-role key and clean up in `afterAll`. They need `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, and `SUPABASE_SERVICE_ROLE_KEY` in the environment.

Required env vars (`.env` for dev, host env for prod):

- `VITE_SUPABASE_URL` / `SUPABASE_URL`
- `VITE_SUPABASE_PUBLISHABLE_KEY` / `SUPABASE_PUBLISHABLE_KEY` (anon key, ships to the client)
- `SUPABASE_SERVICE_ROLE_KEY` (server only ‚Äî used by `client.server.ts` and tests)

## Architecture

### Stack and routing

- **TanStack Start** (file-based routing, SSR-able) on top of Vite 7, deployed to Cloudflare Workers (`wrangler.jsonc`, `@cloudflare/vite-plugin`).
- `src/routes/` is the file-based router; `src/routeTree.gen.ts` is generated by `@tanstack/router-plugin` (do **not** hand-edit).
- `src/router.tsx` wires `routeTree` + a default error component.
- Root layout `src/routes/__root.tsx` sets `<html lang="he" dir="rtl">`, loads Rubik from Google Fonts, mounts `AuthProvider` and `Toaster` (sonner, `position="top-center"`, RTL).
- `src/routes/parent.tsx` and `src/routes/child.tsx` are role-gated layout routes ‚Äî they read `useAuth()` and redirect to `/login` or `/` if the role doesn't match. All real screens live under `/parent/*` and `/child/*`.

### Supabase as the only backend

- Live project: **`flxhxmrtdqegfsupvvus`** (see `supabase/config.toml`).
- `src/integrations/supabase/client.ts` ‚Äî anon client used in the browser. Wrapped in a `Proxy` so `createClient` is called lazily (avoids errors during SSR if env vars aren't materialized yet).
- `src/integrations/supabase/client.server.ts` ‚Äî **service-role** client; only import from `*.server.ts` files (TanStack Start's `importProtection` in `vite.config.ts` blocks client-side imports).
- `src/integrations/supabase/auth-middleware.ts` ‚Äî `requireSupabaseAuth` middleware for `createServerFn`: validates the `Authorization: Bearer` header via `supabase.auth.getClaims`, then injects a RLS-respecting `supabase` client + `userId` into the handler context.
- `src/server/create-child.ts` ‚Äî the one server function: creates a child auth user via admin SDK (`supabaseAdmin.auth.admin.createUser`). A DB trigger `on_auth_user_created ‚Üí handle_new_user` reads `role`, `household_id`, `full_name`, `birthdate` from `user_metadata` and atomically creates `user_roles` + `child_profiles` (a malformed/out-of-range birthdate is clamped to NULL, never blocks signup). **Never** try to insert rows for the child from the client.
- `src/hooks/use-auth.tsx` ‚Äî single `AuthProvider`. Exposes `{ isAuthenticated, isLoading, user, session, role, householdId, childProfileId, signOut, refreshRole }`. `fetchRole` is wrapped in `setTimeout(0)` inside `onAuthStateChange` to release the auth lock before issuing queries.
- `src/integrations/supabase/types.ts` is **generated** from the live DB. Regenerate with the Supabase MCP `generate_typescript_types` tool or `supabase gen types typescript --project-id flxhxmrtdqegfsupvvus` after any schema change.

### Database (live schema)

> Most local files in `supabase/migrations/` were historically aspirational and have been deleted. The repo now contains only `20260518185128_add_savings_and_goals.sql` (the migration that backfills the savings/goals feature on top of the live schema). The earlier migrations that _actually_ shipped (`add_rls_policies`, `fix_handle_new_user_trigger`, `create_household_settings`, `swap_signup_trigger_to_handle_new_user`) live only in Supabase's tracking table ‚Äî bootstrap a fresh env with `supabase db pull`.

**Tables** (text-based status/type fields throughout; no Postgres enums):

- `households(id, name, created_at)`
- `user_roles(id, user_id, household_id, role text CHECK IN ('parent','child','admin'))`
- `child_profiles(id, user_id, household_id, display_name, current_balance numeric default 0, birthdate date, avatar text)` ‚Äî `current_balance` is a **cached wallet balance** maintained by RPCs (not derived on read). `birthdate` (nullable, CHECK 2000-01-01..today) drives the quiz difficulty tier; NULL falls back to the middle tier. `avatar` (nullable text) stores the parent-chosen `"<iconKey>|<colorKey>"` from the catalog in `src/lib/avatars.ts`; NULL falls back to a deterministic icon+color derived from the child id (see `parseAvatar`). Set via `handle_new_user` (from create-child metadata) or `set_child_avatar`.
- `tasks(id, household_id, child_id, created_by_parent_id, title, description, reward_amount, status text CHECK IN ('assigned','submitted','approved','rejected'), submitted_at, reviewed_at)`
- `transactions(id, household_id, child_id, type, amount numeric, reference_task_id, goal_id, reference_quiz_attempt_id)` ‚Äî `type` text CHECK IN `('task_reward','manual_adjustment','goal_allocation','savings_credit','wallet_debit','goal_credit','quiz_reward')`. There is **no `idempotency_key`, no `created_by`**, no enum. The legacy `goal_allocation` value is permitted but unused.
- `household_settings(id, household_id unique, savings_percentage int 0..100 default 10, quiz_subjects text[] default '{}', quiz_reward_amount int 0..1000 default 5)`
- `goals(id, household_id, child_id, title, target_amount int, cycle_amount int, cycle_period text IN ('day','week','month'), status text default 'active', created_by, created_at, updated_at)`
- `quiz_attempts(id, household_id, child_id, subject text IN ('english','math','torah','finance'), total int, correct int, passed bool, paid bool default false, created_at)` ‚Äî one row per quiz finish; `paid` is true only for the first successful attempt per subject per child per Asia/Jerusalem day. Writes happen only via `complete_quiz_and_pay`.

**No `has_role()` or `get_user_household_id()` helpers exist on this DB** ‚Äî RLS policies inline `SELECT household_id FROM user_roles WHERE user_id = auth.uid()` instead.

**RPCs:**

- `approve_task_and_pay(p_task_id uuid) RETURNS boolean` ‚Äî locks the task, idempotent on "transactions exist for this task". Inserts `task_reward (+full)`, and if `household_settings.savings_percentage > 0` also inserts `wallet_debit (-save)` + `savings_credit (+save)`. Updates `child_profiles.current_balance` to reflect wallet delta only.
- `deposit_to_goal(_goal_id uuid, _amount integer) RETURNS jsonb` ‚Äî child-owner or household-parent. Validates wallet ‚â• amount and `deposited + amount ‚â§ target`. Inserts `wallet_debit (-amt)` + `goal_credit (+amt)`, updates cached balance, flips goal to `completed` when target reached. Returns `{ success: true, deposited, target }` or `{ error: '...' }`.
- `set_child_birthdate(_child_id uuid, _birthdate date) RETURNS jsonb` ‚Äî household-parent-only edit of `child_profiles.birthdate` (range-validated, NULL allowed to clear). Used by the edit dialog on `/parent/children`; deliberately an RPC rather than a client-side UPDATE so parents never get a broad UPDATE path to `current_balance`.
- `set_child_avatar(_child_id uuid, _avatar text) RETURNS jsonb` ‚Äî household-parent-only edit of `child_profiles.avatar` (empty string clears to NULL). Same edit dialog on `/parent/children`, same reason for being an RPC as `set_child_birthdate`.
- `complete_quiz_and_pay(_subject text, _correct int, _total int) RETURNS jsonb` ‚Äî called by the authenticated child after a quiz. Records the attempt; pass = ‚â•80%. Pays out only the first passing attempt per subject per Asia/Jerusalem day: inserts `quiz_reward (+reward)`, applies the savings split exactly like `approve_task_and_pay`, updates `child_profiles.current_balance`. Returns `{ passed, paid, wallet_delta, savings_delta, reward, attempt_id }` on payout, `{ passed: false, paid: false, reason }` on fail or already-paid, or `{ error: '...' }`.

**Storage:** there is no `task-proofs` bucket in this DB. The plan envisioned proof-image uploads, but the live schema doesn't have a `proof_image_path` column on tasks and no Storage bucket has been provisioned. Task approval currently doesn't require proof.

### Wallet vs Savings balance ‚Äî single ledger, derived

`transactions` is the only source of truth. Both the RPCs and any read-side code should compute balances by **filtering on `type`**:

```text
wallet_balance   = SUM(amount) WHERE child_id = X AND type IN ('task_reward','manual_adjustment','wallet_debit','quiz_reward')
savings_balance  = SUM(amount) WHERE child_id = X AND type = 'savings_credit'
goal_deposited   = SUM(amount) WHERE goal_id = G AND type = 'goal_credit'
```

The canonical wallet-tx allowlist lives in `src/lib/transactions.ts` (`WALLET_TX_TYPES` / `isWalletTx`) ‚Äî reuse it instead of hardcoding the list in each route.

`child_profiles.current_balance` is a cached mirror of `wallet_balance` updated by `approve_task_and_pay`, `deposit_to_goal`, and `complete_quiz_and_pay`. Anything that mutates the wallet outside those RPCs (e.g. a future manual-adjustment flow) must keep the cache in sync.

> Why both `wallet_debit (-amt)` and `goal_credit (+amt)` on a single deposit? They live in different "pots" but share one ledger table. Wallet sum excludes `goal_credit`; goal pot only counts `goal_credit`. Don't add `goal_credit` to the wallet formula or deposits will appear to net to zero.

### Styling

- Tailwind v4 via `@tailwindcss/vite` (CSS-first config in `src/styles.css`, no `tailwind.config.js`).
- shadcn/ui-style primitives in `src/components/ui/*` (Radix wrappers). `components.json` is configured; new shadcn additions land here.
- App-level components: `coin-amount.tsx`, `child-avatar.tsx`, `status-badge.tsx`, `loading-skeletons.tsx`.
- **RTL discipline**: layouts use logical sides (`ms-`, `me-`, `start-`, `end-`) ‚Äî never `ml-`/`mr-`/`left-`/`right-`. All copy is Hebrew.
- The shadcn `Progress` primitive uses CSS `translateX` which fills from the left edge regardless of `dir="rtl"`. For directional progress bars in this RTL app, use a plain `<div style={{ width: '${pct}%' }}>` ‚Äî it inherits the start edge from its container (see `/child/savings` for an example).

## Conventions

- **Never** modify Supabase schema, RLS, or RPCs without an explicit ask ‚Äî there is a single Supabase project shared with the mobile app. Use the Supabase MCP `apply_migration` after asking for confirmation, then regenerate types in the same turn.
- All UI copy in Hebrew; layouts must stay RTL-safe (no `left-`/`right-`/`ml-`/`mr-`).
- Money mutations go through SECURITY DEFINER RPCs (`approve_task_and_pay`, `deposit_to_goal`); never insert rows into `transactions` from the client and never let `child_profiles.current_balance` drift from the wallet ledger.
- Server-only code lives in `*.server.ts(x)` files; the `tanstackStart` plugin will error if those are imported from client modules.
- `src/routeTree.gen.ts` is regenerated on dev ‚Äî never hand-edit.
- After any DB change: regenerate `src/integrations/supabase/types.ts` so the rest of the codebase compiles against reality.
